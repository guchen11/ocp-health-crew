Root cause analysis

CSI clone chain: working vs broken

The snapshot clone strategy builds a 4-level chain per VM disk from one source image. When the temp snapshot on the source is removed while Kubernetes still believes provisioning succeeded, Ceph cannot resolve parent references. 11 / 10,038 clones (0.1%) showed this under extreme single-image contention.

Broken chain (RCA)

The foundation (temp snapshot on source) was deleted. Downstream objects may still exist in the pool but are not openable.

Race condition mechanism

Heavy contention on one source image with ~10K concurrent clones serializes work per image. GRPC / provisioner time bounds interact with slow Ceph completion.

    autonumber
    participant P as CSI provisioner
    participant C as ceph-csi
    participant R as Ceph single source image

    P->>C: Start clone CreateVolume
    C->>R: Create temp snapshot on source begin clone chain
    Note over R: Heavy queue clones take minutes to hours
    P-->>C: Timeout or slow return e.g. 1800s boundary
    Note over P: Assumes failure triggers cleanup
    C->>R: Delete temp snapshot on source
    Note over R: Temp snap removed foundation gone
    R-->>C: Clone objects may already exist async
    Note over P: PVC Bound DV Succeeded K8s thinks OK
    Note over R: Chain exists but root snapshot missing Ceph cannot open images

Under normal load the window is small (clones finish in seconds). Under 10K single-image contention the window grows enough for a small fraction of operations to hit the race.

Evidence (summary)

  • Source snapshot count: ~285 active temp snapshots on the source for successful clone batches; none remaining for the 11 broken cases.
  • Broken -temp images: still listed in Ceph but fail with parent not found when opened.
  • Working clones: final image to temp image to temp snapshot on source forms a valid parent walk.
  • Kubernetes state: PVC Bound, DataVolume Succeeded – control plane does not reflect Ceph chain health.
  • VMs: 11 VMs stuck 7–34 hours with the same attach error – consistent with permanent chain corruption, not a transient retry issue.
